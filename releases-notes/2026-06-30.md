# Release Notes - 2026-06-30

## Highlights

This release focuses on PrecisionProDB SQLite-mode performance, especially high-sample `ALL_SAMPLES` population mode. The main result is a large reduction in runtime while preserving byte-identical outputs across the benchmarked datasets.

### SQLite Full-Run Improvements

The original SQLite full-run profile, using a large mutation input and a prebuilt GENCODE SQLite annotation database, identified variant-to-protein mapping as the primary bottleneck:

- Original total: `252.756s`
- Mapping stage: `101.525s`, about `40.2%`
- SQLite query time itself was only `5.664s`

After optimizing interval lookup in `buildSqlite.py`, the full PrecisionProDB run improved:

```text
252.76s -> 168.68s
```

Then vectorizing `new_nt` assignment in `perChrom.py` improved the same full-run path further:

```text
168.68s -> 125.96s
```

All main outputs were byte-identical after optimization:

- `*.pergeno.aa_mutations.csv`
- `*.pergeno.protein_changed.fa`
- `*.pergeno.protein_all.fa`

## Population Mode Performance

The population-mode benchmark used pre-split chr19-22 mutation TSV inputs with:

```text
SQLite: prebuilt GENCODE annotation database
datatype: GENCODE_GTF
individual: ALL_SAMPLES
threads: 20
force_memmap: True
```

Initial profile:

```text
total wall time: 15:14.70
translation tasks: 356,183
dominant stage: translation_pool_collect, 694.92s
```

Final benchmark after the current optimization series:

```text
total seconds: 299.323
wall time: 5:00.13
max RSS: 12.69 GB
exit status: 0
```

This is about a `67%` runtime reduction for the chr19-22 population-mode benchmark.

## Major Code Changes

### `src/precisionprodb/vcf2mutation.py`

- Rewrote `tsv2memmap` to avoid pandas batch loading for simple `0/1` TSV genotype matrices.
- chr19 TSV memmap benchmark improved:

```text
10:56.22 -> 0:15.33
```

### `src/precisionprodb/buildSqlite.py`

- Added batch interval lookup for variant-to-protein mapping.
- Replaced repeated full-DataFrame boolean filtering with sorted numpy arrays and bounded candidate windows.
- Full SQLite run improved:

```text
251.05s -> 168.68s
```

### `src/precisionprodb/perChromSqlite.py`

- Reduced multiprocessing payload for translation workers.
- Passed shared mutation data once per worker via pool initializer.
- Kept backward-compatible wrapper behavior.

### `src/precisionprodb/perChrom.py`

Main population-mode translation optimizations:

- Added worker-local CDSplus caching.
- Added indexed CDSplus/mutation merge fast path.
- Added direct merge bypass for common direct-translation cases.
- Vectorized `new_nt` assignment.
- Removed pandas-heavy same-length variant formatting.
- Optimized codon lookup and complex codon translation:
  - replaced `getCodons(...)` pandas `groupby(...).agg(...)`;
  - replaced complex-path `AA_alt apply(axis=1)`;
  - added codon translation cache.
- Added mutation record caching in `getMut_helper(...)`.
- Added a conservative sparse direct-translation path for clean one-base substitutions.

## Profiling Conclusions

The bottleneck shifted during the optimization work:

1. Initial population mode was dominated by translation after sample grouping.
2. Early worker-internal profiles showed direct-path pandas merge/new-nt construction and complex codon handling as the main costs.
3. After direct merge bypass and codon-path optimization, worker CPU dropped sharply:

```text
worker_task_total: 4169.123s -> 1859.505s
worker_complex_getCodons_alt: 959.347s -> 65.591s
worker_complex_getCodons_ref: 617.550s -> 53.563s
worker_getMut_helper: 806.277s -> 171.356s
```

4. Final profiling indicates remaining costs are more distributed:
   - direct input construction;
   - direct translation result generation;
   - fallback merge handling;
   - fixed parent-side costs such as SQLite init and population grouping.

At this point, additional optimizations are expected to have smaller returns unless they change the direct-path data model or parent-side scheduling.
